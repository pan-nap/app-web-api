# 核心原则与设计模式

## 核心原则

**数据与页面完全解耦**：所有数据获取、修改、处理操作都在Store层完成，页面层只负责调用Store方法和UI展示。

## 设计模式

### Store层职责
- **接口调用**：所有API请求都在Store中进行
- **数据处理**：数据解析、转换、封装在Store中完成
- **状态管理**：维护应用状态和数据缓存
- **业务逻辑**：复杂的业务逻辑在Store层实现

### 页面层职责
- **UI展示**：只负责页面渲染和用户交互
- **方法调用**：调用Store提供的公开方法
- **事件处理**：响应用户操作，调用对应的Store方法

## 优势

1. **代码复用**：Store方法可在多个页面中复用
2. **易于测试**：Store层可独立进行单元测试
3. **维护性强**：数据逻辑集中管理，修改时只需关注Store层
4. **职责清晰**：页面层和Store层职责分明，降低耦合度
5. **扩展性好**：新增功能时只需在Store中添加相应方法

## 使用场景

- 用户登录/退出及相关数据获取
- 字典数据、配置信息等全局数据管理
- 复杂的业务逻辑处理
- 需要在多个页面共享的数据状态
- 需要缓存的数据管理
